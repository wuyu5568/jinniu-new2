# 金牛协议（认购与分红）

用户以认购金额开立认购订单，按日获得静态收益；推荐与社区激励产生每日分红。两类收益均可推进订单出局进度，但只有认购金额计入本金。

**MVP（第一期）**：四期交付中的第 1 期——用户（钱包 eth_authorize 登录；首次须填邀请码）、认购下单、订单级静态收益（含提取掉头）、可提余额与 6% 提取；管理 API 与用户 API 同仓库同进程、分目录。第 2 期代数分红；第 3 期社区 V 级 / 级差 / 平级 / 断档；第 4 期对接既有 **dapp-admin**；链上充值见 ADR 0007；**提取链上打款**见 ADR 0010（默认关，开后热钱包打到账额）。

## Language

**认购金额**:
用户开立认购订单时投入的本金（U）。出局倍数按该金额所在档位计算。
_Avoid_: 质押金额、投资额、本金（对外口径）

**认购订单**:
一笔认购形成的订单，记录认购金额、出局倍数、已累计收益与状态（进行中/已出局）。代码与持久化沿用标识 Location / locations，语义等同本词。
_Avoid_: 质押订单、仓位

**静态收益**:
认购订单按当日静态收益率释放的收益，计算公式为 **认购金额 × 静态收益率**；出局目标仍为 **认购金额 × 出局倍数**（累计满额出局，与日息基数分离）。**双记**：增加该单出局累计，并增加用户可提余额。
_Avoid_: 日息、自然释放、挖矿收益（对外口径）；按出局金额计息（已废止，见 ADR 0011）

**静态收益率**:
挂在**认购订单**上：每笔订单开单第 1 天为 0.6%，之后在 0.6%–1.4% 间按日步进 0.05% 升降；与同用户其他订单相互独立。
_Avoid_: 日利率、释放比例、用户级统一利率（以及错误写法「0.5% 步进」）

**每日分红**:
推荐/社区激励产生的收益类型统称。**代数奖励、社区基础奖、平级分红**按 FIFO 填入本人进行中认购单：增加出局累计，并同步增加可提余额（双记）；无进行中订单则当日该笔作废，填满全部进行中单后的剩余亦作废（不作废部分以外的金额不进可提）。
_Avoid_: 动态收益；无单仍发可提；超额部分仍进可提

**账户余额**:
用户可用于认购的中心化账本余额；来源含：管理端充值（增加）、管理端修改账户余额（设目标值）、以及链上充值——用户通过 **BSC 主网分账充值合约**（`0x0f299470…`，`buy(num)`）入账；进程内按 `app.deposit_cron`（默认每分钟）自动按合约序号拉取，管理端 HTTP 亦可手动触发；**≥5 USDT** 且已注册才入账；未注册或低于最低额仍推进游标并记跳过（`eth_user_record` 一序号一行，`UNIQUE(last)`）；补注册后可按序号重放 skipped 入账。链上充值按用户 `buy` **全额**记入账户余额，与资金在链上如何分到各收款地址无关。与可提余额分开记账。
_Avoid_: 钱包余额、可提余额；仅创世 JWT 充值；每用户独立充值地址；按分账比例给用户少记；USDT Transfer 扫块 / eth_getLogs；Webhook/独立扫块进程；未注册卡住后续序号；`<5` 中断整批拉取；强制最低链上充值额 100（已改为 5）；无唯一游标导致重复入账；跳过不推进游标；已跳过序号靠普通拉取自动补账；仍指向旧分账合约 `0x1E13586a…` / `0xb8f765…` 或 BuySomething `0x49c735…` 作为唯一入口；仅依赖人工点「拉链上充值」才入账

**充值分账**:
用户每次链上充值时，合约将收到的 USDT **立刻**按 owner 配置的收款地址列表与百分比（基点，合计 10000）拆转到各方；名单由 owner **整表替换**，最多 5 个地址。分账只决定资金链上去向，不改变用户账户余额入账口径。合约对外保持与既有充值合约同型的 `buy` + 序号查询，便于按序号入账。
_Avoid_: 每用户独立充值地址；资金长期留在合约再人工归集；按分账比例给用户入账；单条增删改收款人而不整表替换；整数百分比无法表达 0.5% 档

**可提余额**:
用户当前可提取的资产余额；静态收益与每日分红（代数/社区/平级）入账时增加（分红仅就可填进出局空间的部分），提取资产时扣减；管理端可修改为目标值（差额记管理端调整可提流水）。不可划回账户余额；再认购须重新充值账户余额。
_Avoid_: 钱包余额（链上含义）、账户余额（认购用）；可提划转认购；无单/超额分红仍进可提

**管理端调整可提**:
管理端将可提余额设为目标值时产生的账本流水类型；金额为差额（可正可负）。不计入用户收益、不进入账户充值列表。
_Avoid_: 充值、提取、静态收益

**出局**:
认购订单的累计入账（静态收益 + 每日分红）从 0 起加总，达到「认购金额 × 出局倍数」时订单结束、停止继续释放。
_Avoid_: 退出、清仓、爆仓；开单时预记认购金额再累加

**出局倍数**:
由认购金额档位决定：100–999U 为 2 倍，1000–2999U 为 2.5 倍，≥3000U 为 3 倍。
_Avoid_: ROI、退出倍数
    
**提取资产**:
用户从可提余额确认提出指定金额（任意正数即可，无最低额限制）。**须已开启链上打款且热钱包配置齐全**，否则拒绝（不扣可提、不掉头）。确认时立刻：扣减可提、按费率锁定手续费与到账额、对该用户全部进行中认购订单做**提取利率掉头**、建单为待打款（`rewarded`），并**异步**发起链上打款。无管理端审核；确认后不可取消。打款中为 `doing`，成功为 `pass`。打款失败不自动退可提，可由管理端重试打款。代码/RPC/表贯穿标识 Withdraw / withdraws。详见 ADR 0012、ADR 0018。
_Avoid_: 提现（可作同义，优先用提取资产）；待审 pending 审核流（已废止）；用户取消已确认单；申请即视为链上到账；打款关闭时仍扣可提；强制最低提取额；免手续费；对已出局订单掉头

**提取利率掉头**:
提取资产确认时，对全部进行中认购订单逐笔处理：方向为 **down** 的不掉头；无**昨日结算利率**（从未静态结算或字段为空）的不掉头；其余 **up** 且本结算周期尚未掉头的：以昨日结算利率为起点——若昨日已是最低档（默认 0.60%），则把订单改回 **最低档 / up**（今天仍按最低档计息）；否则按该昨日档 `AdvanceRate(down)` 一档写入（例：昨日 0.70% → 提现后 0.65% / down）。下次静态结算按写入后的利率计息，发完后再照常步进，并解除本周期掉头标记。昨日结算利率在每次静态计息时写入订单。同周期内多次提取不重复掉头。已出局订单不参与。
_Avoid_: 提现从「日结后已步进」的当前档起步；提现只翻方向不改利率；无昨日利率仍强行掉头；昨日最低档时 down 步进触底反弹；同周期多次提取反复翻转；对已出局订单掉头

**提取打款**:
将已审待打（`rewarded`）的提取单，由服务端热钱包在 BSC 上向用户地址转 USDT（金额为到账额）；广播后先记交易哈希再确认上链，成功后为 `pass`。可由定时任务与管理端触发。详见 ADR 0010。
_Avoid_: 审核通过同步强制上链且无独立状态；无私钥管控的多实例并行打同一单

**用户**:
以钱包地址为身份标识；登录时对**地址本身**做 personal_sign（无 nonce）；服务端验签后签发 JWT。首次登录须在同一次登录请求中提交邀请码并绑定邀请人（创世账号除外）。
_Avoid_: 用户名密码登录；先验签再另接口绑邀请码；本期强制 nonce 挑战（已有 v1 接口保留，兼容前端 eth_authorize 走无 nonce）

**激活用户**:
至少开立过一笔认购订单的用户（含已出局）。看板 `totalUser` = 激活用户数；`todayUser` = 当日首次开立认购订单的用户数（上海自然日）。注册未认购不算激活。
_Avoid_: 未锁定用户、未软删用户、仅进行中认购才算激活

**登录挑战**:
可选能力：v1 接口仍可下发一次性 nonce；当前用户端兼容登录（eth_authorize）不使用挑战，直接对地址签名。
_Avoid_: 把 JWT 当成登录挑战；本期前端主路径依赖 nonce

**邀请码**:
首次登录时必填；值为**已注册用户的钱包地址**。校验通过后建立与邀请人的推荐关系，方可完成注册/登录。创世账号除外。
_Avoid_: 独立邀请口令、随机短码（非本项目口径）

**邀请人**:
邀请码所对应的已注册用户；成为新用户的直属上级（推荐关系）。**绑定后永久不可更改**。
_Avoid_: 推荐人（可作同义，优先用邀请人）；可改上级

**直推**:
邀请关系上的直属下级（仅注册即可形成）。用于拆腿与社区树结构；**不**单独作为代数解锁与对外「推荐人数」的计数口径。
_Avoid_: 与有效直推混用

**有效直推**:
直推中至少有过一笔认购订单（含已出局；订单记录被删除则不再计入）的用户；软删除不取消有效直推资格。代数解锁代数、以及对外展示的推荐人数，均按有效直推计数；规则变更不回溯已发代数。推荐名单仍可列出全部直推，未激活者可标注未激活。
_Avoid_: 仅注册未认购也算；仅进行中订单才算；按累计金额另设门槛；名单隐藏未认购直推；追回历史多发代数；软删即取消激活

**创世账号**:
预置的根钱包用户；可不填邀请码即可完成首次登录/注册，作为推荐树顶点。地址由配置或初始化写入。
_Avoid_: 管理员（权限角色）、普通邀请人

**业务时间**:
持久化 `DATETIME` 与管理端展示均按**北京时间墙钟**（Asia/Shanghai）；日结自然日亦按上海。历史 UTC 墙钟已一次性 +8 小时迁移（见 ADR 0019）。
_Avoid_: 库内 UTC 墙钟与展示混用；对 `settle_date` 做时区平移

**静态收益结算**:
对进行中认购订单按日入账静态收益并推进次日收益率（含提取已立刻步进的订单：按订单当前档计息再步进）。正式环境由进程内 cron 每日定时执行（可配置表达式与时区，默认上海），管理端可手动触发。同日后续顺序为：代数奖励 → 刷新社区等级 → 社区基础奖与平级分红（同次计算、分流水入账；奖源为全部下级树，含大区）。以 `settle_runs` 按上海自然日 **INSERT 占位**防重（多实例安全）：占不到则跳过；占位后中途失败不自动重跑。`force=1` 仅当配置允许（`allow_force_settle`）时可强跑（可能重复入账，仅测试）。
_Avoid_: 仅懒结算、无定时任务；先按旧等级发社区奖再刷新等级；生产环境开启 force；多实例先查再写而不占位；提取掉头后结算仍按旧档计息；独立二次遍历上行平级；社区奖结算剔除最大腿

**管理 API**:
与用户 API 同仓库、同一服务进程，路径前缀 `/api/admin_jinniu`；管理端账号密码登录后发 JWT，用于充值、查数、触发结算等运维操作。
_Avoid_: 与用户接口混在同一巨型 proto；MVP 独立管理微服务；admin_key 明文头鉴权（已删除）

**管理 Web**:
运维界面使用既有仓库 **dapp-admin**（不拷进本仓）；覆盖充值、用户/订单、结算、邀请关系、业务参数调整等。
_Avoid_: 本期在本仓新建 Vite+React 管理端；密钥长期存 localStorage

**API 列表响应**:
列表类 HTTP 接口响应体为裸 JSON 数组 `[{...},...]`；分页 token 放在响应头 `X-Next-Page-Token`。收益按日汇总另走 `/earnings/days`。单资源（资料、充值结果、看板、结算结果等）仍为 JSON 对象。
_Avoid_: 列表再包一层 `{"users":[...]}`；把分页 token 塞进数组元素

**用户 Web**:
同仓库 `web/app`（Vite + React）；登录仅 MetaMask：连钱包 → 查是否注册 → 未注册填邀请码再签名；JWT 存 sessionStorage；规划增补下级树邀请关系、每日收益日历。
_Avoid_: 登录页私钥入口；先签名再因缺邀请码失败

**每日收益明细**:
当日入账流水，类型含静态收益、代数奖励、社区基础奖、平级分红（提取与充值不计）。
_Avoid_: 仅静态；把提取/充值算作收益

**业务参数**:
管理端以键值表（`business_configs`）逐项调整并写入 MySQL；进程内聚合为业务参数对象热更新，立即用于后续业务，不回溯历史。含：提取手续费；代数比例与最大代数；平级比例与最低 V 级；静态利率上下限与步进；最低认购/提取金额；认购档位；出局倍数三档；社区 V1–V9 门槛与比例。创世地址、JWT/管理密钥等仍在配置文件。
_Avoid_: 仅改 config.yaml；改参后重算历史；在页面改 JWT/管理密钥；以整包 JSON 为唯一编辑入口

**认购档位**:
管理端可配置的一组预设认购金额（如 100 / 500 / 1000 / 3000），供用户端快捷选择；用户亦可自填任意金额，但不得低于最低认购金额。档位本身不决定出局倍数（出局倍数仍按金额区间）。
_Avoid_: 管理端代用户认购；档位与出局倍数档位混为一谈；仅允许点选档位不可自填

**管理用户数据**:
管理端可对用户主档与认购订单做增删改查。邀请人**创建后不可改**。删除为双接口：默认软删除；硬删除须无下级、无认购订单、余额均为 0，前端二次确认。
_Avoid_: 管理端改邀请人；有下级/订单/非零余额时硬删；仅硬删无软删

**代数奖励**:
每日分红的一种：按下级当日静态收益的 5% 计发，最多 10 代；**有效直推** 1 人解锁第 1 代，有效直推 n 人解锁前 n 代（n≤10）。结算静态时同步发放；FIFO 双记（出局累计+可提），无进行中单或超额部分作废。
_Avoid_: 按仅注册直推解锁代数；直推奖（仅一代时）、级差奖；无单/超额仍进可提

**社区等级**:
按小区业绩对照 V1–V9：V1≥5千/10%，V2≥2万/20%，V3≥8万/30%，V4≥25万/35%，V5≥50万/40%，V6≥150万/45%，V7≥500万/50%，V8≥1000万/55%，V9≥2000万/60%；取满足门槛的最高档。管理端手设级别会**锁定**；锁定期间结算只更新小区业绩、不改级别；管理端可单独解锁，解锁后下次结算起按业绩重算。
_Avoid_: VIP；自定义另一套门槛/比例（非本约定）；手设后仍被结算覆盖

**社区基础奖**:
按**级差**计算（与平级分红分离）：在领取人**全部直推下级树**（含大区与小区；仅一条腿时该腿也计）内，对每个有当日静态的节点：若路径上（自奖源的**上级**起）存在**高于或等于**领取人的 V 级则断档——**只跳过该同级/更高节点之下的下级**，该节点**本人**仍计级差；对照比例为奖源与领取人之间（**不含两端**）节点中 **V 级最高** 的那一档比例（没有中间人则为 0%），奖额 = `静态 × (本人 V 级比例 − 对照比例)`，比例差 ≤ 0 不计。**例外：直推本人**当日静态不走对照/断档，一律 `静态 × 领取人当前 V 级比例`（含直推同级/更高）。与直推相关的平级可叠发。与静态同日结算；FIFO 双记（出局累计+可提），无进行中单或超额部分作废。定级仍只看小区业绩，与发奖范围无关。
_Avoid_: 按历史累计业绩每天全额乘比例；对照用最靠近领取人/最靠近底层；整棵子树静态不经去重直接乘；同级改发平级而不算级差；同级/更高节点本人也一律断档为 0；对照含奖源本人等级；全局统一 calcBase × V 比例；按认购金额而非静态计社区奖；无单/超额仍进可提；发奖时剔除最大腿/大区；直推同级时连直推本人也不发全额；直推全额级差与平级互斥

**平级分红**:
与社区基础奖分开计算。先算完全部社区基础奖后：仅看**直推上级**一层——若来源用户当日社区基础奖合计 > 0、直推上级等级 **≤** 来源用户等级、且**双方 ≥ 平级最低档**（默认 V3），则该直推上级获得 `来源当日社区基础奖合计 × 平级比例`（默认 10%）。不要求等级完全相同；不隔代查找；可与直推全额级差叠发。基数**只含社区基础奖**，不含平级本身（不连环抽）。规则变更不回溯历史。入账 FIFO 双记（出局累计+可提），无进行中单或超额部分作废。
_Avoid_: 平级按本人静态 × 10%；同级从级差改道；对平级再抽平级；必须完全同级才发；沿链找最近同级/最近≤级；取消 V3 门槛；与直推全额级差互斥；追回/补发历史；无单/超额仍进可提；直推同门基础奖建池均分（已废弃）

**业绩**:
某范围内**下级**用户的历史累计认购金额之和（含已出局；不因出局扣减）；**不含**该范围根节点本人的认购。
_Avoid_: 仅进行中认购合计；出局目标合计；把本人认购算进团队业绩

**小区**:
按**直推**（注册关系）拆腿：每个直推及其整棵下级树为一条腿，不论是否有效直推；**小区业绩** = 去掉最大腿后其余各腿业绩之和。用于对照 V1–V9 定级。
_Avoid_: 仅按有效直推拆腿；整户合计直接定级（非本约定）；只看最大腿定级

**腿**:
某一个直推出发的整棵下级树；社区基础奖的断档按「断档节点及其子树」切分，不是整腿一刀切。
_Avoid_: 与小区混称；断档即整腿停发社区奖；未认购直推就不形成腿

**断档**:
对社区基础奖：若奖源之上的路径上出现社区等级**高于或等于**领取人的节点，则**仅该节点之下的下级**当日静态不计入领取人级差；该同级/更高节点**本人**仍走正常级差（**直推本人全额比例例外**仍优先）。同腿上断档点之上、或其他分支仍可计。代数分红不受影响。平级仅看直推上级且允许上级等级≤下级，**不适用**该断档拒发。
_Avoid_: 超越奖；同级/更高节点本人也一律断档为 0；整户一刀切停社区奖；断档时停代数；断档后整条腿停发；直推同级/更高时连直推本人全额也不发；平级仍按断档子树拒发
**停分红**:
管理端开关。开启后该用户仍可登录、仍拿本人静态与分红；但其当日静态**不再作为奖源**——上级不从他拿代数、社区基础奖、平级。
_Avoid_: 与软删（禁登录）混用；停本人静态结算
